# PHANTM Functionality Audit

The current build has a polished visual shell, but several core interactions are too shallow or unreliable. The canvas navigation depends on a long-press state change before the root pan responder captures movement, which can fail because responder ownership is decided before the long-press activation path. Ring slot buttons are visually present but small and easy to miss, while no fallback module launcher exists for users who cannot reliably hit the orbital targets. The Nest Editor advertises module slot configuration, but the module list is display-only and does not write slot changes back to `nest.rings`. The create-nest action creates a generic nest without asking for a label or making the new nest active, so it looks like a gesture button but does not produce a clear flow. Hidden nest unlocking sets the active nest but should be paired with canvas navigation and clear feedback. Module routes exist, but the main functionality should be reachable from the canvas without relying on hidden or tiny touch targets.

The repair pass should preserve the frontend-inspired PHANTM design while making functionality explicit and reliable. The main canvas needs direct module-launch controls, a clear add-nest flow, reliable directional movement controls, visible active-nest switching, and a functional Nest Editor that can assign modules or nest links to ring slots. The state store needs helper methods that update slot configurations and actively focus newly created or imported nests. Tests should cover nest creation, slot replacement, active nest focusing, notes/captures persistence actions, import behaviour, and utility logic.
